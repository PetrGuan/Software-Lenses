# Boyd Mindset

> "The side that orients and adapts faster wins."

**Category:** Strategy | **Skill:** `/boyd`

## When to Use

- Incident response is slow despite good tools
- Teams collect data but react late
- Product cycles miss learning windows

## Core Move

Map OODA latency and quality; remove friction between observation and action.

## Diagnostic Questions

- Where does signal-to-decision time stall?
- Are teams orienting with outdated mental models?
- Which decision gates add delay without reducing risk?

## Output Format

```
OODA Map:
  [observe -> orient -> decide -> act]

Latency Bottlenecks:
  [where cycle time is lost]

Orientation Gaps:
  [wrong assumptions]

Loop Improvements:
  [instrumentation, authority, runbook changes]
```

## Anti-Patterns This Catches

- Rich dashboards that no one acts on because the decision path is unclear
- Incident post-mortems that identify root cause but never shorten response time
- Decision gates that add latency without adding information or safety
- Teams that orient on stale mental models and misread new signals
- Slow escalation chains that let small incidents become large outages

## Pairs Well With

- **Shannon** — Shannon improves signal quality, Boyd ensures the signal reaches a decision-maker fast enough to matter
- **Deming** — Deming improves the system that produces failures, Boyd improves the system that responds to them
- **Simon** — Simon sets the threshold for when to act, Boyd compresses the time from observation to that action
- **Feynman** — Feynman diagnoses what is actually happening, Boyd structures the loop that turns diagnosis into response

## When Not to Over-Apply

- The situation calls for deliberate, slow analysis rather than rapid iteration
- Compressing the loop would skip necessary safety or compliance reviews
- The bottleneck is not speed of response but correctness of understanding
